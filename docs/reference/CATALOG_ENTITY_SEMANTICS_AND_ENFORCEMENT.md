# Catalog entity semantics and enforcement

Canonical definitions for the **REFERENCE.CATALOG** vocabulary: what each object means economically, how it links to Snowflake, and where enforcement lives today (dbt seeds/tests vs. warehouse DDL).

---

## CONCEPT ŌĆö What are we measuring, economically?

The **causal vocabulary**. Each row is a named economic phenomenon (rent, absorption, cap rate) that drives or is driven by others. It is **not** a vendor column and **not** a metric row ŌĆö it is the abstract thing a metric measures. Every metric, dataset, and feature should trace to a concept.

**Population / dimensioning (intended):** `direction` Ōłł {positive, negative, neutral}; `metric_category_code` Ōłł {leading, lagging, coincident} via `metric_category`. New datasets should not register without a valid `concept_code` FK.

**In this repo:** `concept.domain` is tested as an FK to `domain.domain_code` (`schema_data_infrastructure.yml`). `direction` and `metric_category_code` have `not_null` + `accepted_values` / `relationships` where applicable.

---

## METRIC ŌĆö How is a concept measured by a specific vendor at a specific grain?

The **physical measurement contract**: one row per vendor ├Ś column ├Ś grain (and related keys). Links a concept to a real Snowflake column and table path.

**Enforcement (intended):** For `is_active = TRUE`, `concept_code`, `vendor_code`, `geo_level_code`, `frequency_code`, `table_path`, `snowflake_column` (and related fields) should be non-null; `geo_level_code` Ōłł `geo_level`; `frequency_code` in the frequency vocabulary; `unit` in a controlled set.

**METRIC_RAW vs METRIC:** `metric_raw.csv` is authoring; `metric.csv` is built (see `docs/reference/METRIC_CSV_BUILD_SPEC.md` and `scripts/reference/catalog/build_metric_csv_from_metric_raw.py`). **Promotion gate:** singular test `tests/catalog/assert_active_metric_raw_promoted_to_metric.sql` ŌĆö every **metric_raw** row with `is_active` true and `data_status_code = active` must appear in **metric** (backlog rows stay non-active in raw until promoted).

---

## DATASET ŌĆö What is the source system and its operational status?

**Pipeline and freshness registry**: one row per logical data product (e.g. ACS demographics at CBSA). Answers whether it is live, where it lives in Snowflake, last refresh, and MotherDuck serving.

**Enforcement (intended):** Every new `serving.iceberg` table should have a `dataset` row with `is_motherduck_served = TRUE` and `last_refresh_date` maintained by the export task.

**In this repo:** **`dataset_product_type`** is the **authoring** bridge; **`dataset.product_type_codes`** is **machine-derived** via `scripts/reference/catalog/sync_dataset_product_type_from_bridge.py` (see **`DATASET_PRODUCT_TYPE_CODES_FROZEN.txt`**). A **follow-up PR** will drop **`product_type_codes`** once consumers use the bridge only. **`last_refresh_date`** for **MotherDuck-served Ōł¦ active** is a **hard-fail singular test** (`tests/catalog/assert_motherduck_active_dataset_has_refresh_date.sql`) plus `column_types: date`. Warehouse nullability remains **DDL-only** ŌĆö **`ddl/catalog_constraints.sql`**.

---

## VENDOR ŌĆö Who produces the data and what are our contractual terms?

**Procurement and trust:** contract status, refresh cadence, data share type, expiry. No `metric` row should reference a missing `vendor_code`.

**In this repo:** **`vendor.is_motherduck_served` was removed** ŌĆö MotherDuck readiness is **dataset-only** (`dataset.is_motherduck_served`, `dataset.last_refresh_date`). **`contract_expiry_date`** is seeded with **`column_types: date`** on the vendor seed config (warehouse DDL may still need alignment).

---

## OFFERING ŌĆö What capital strategy does this serve?

**Investment thesis anchor** (e.g. RED / REQ / RESI). Signals route through **`concept_offering_weight`** (and related offering metadata).

**Dense matrix:** `concept_offering_weight.csv` is generated by `scripts/reference/catalog/sync_concept_offering_weight_from_matrix.py` as **every active offering ├Ś every `concept_code` in `concept.csv`** (including **inactive** vocabulary rows), default weight `0`. That yields **3 ├Ś 34 = 102** rows with the current seeds; previously only **active** concepts were expanded (**31 ├Ś 3 = 93**), which left nine pairs missing for inactive concepts (`multifamily_market`, `spine`, `underwriting`). Those nine rows now exist with weight `0`.

**Offering dimensioning:** `offering.vertical_code` should FK to `vertical` (see seed tests in `schema_offering_tearsheet.yml`).

---

## OPCO ŌĆö Which operating company manages these assets?

**Operational identity:** links properties to management structure; carries `vertical_code`, external flags, parent entity.

**`pretium` row:** Fund-level OpCo ŌĆö **`vertical_code` is intentionally null** (spans RESI / REQ / RED). Documented on the seed row in **`catalog_entity_note`**; do not assign a single vertical that misstates reality.

**Gaps:** Resolve `geo_status_code` semantics (enum vs. drop). `strategy_code` and `primary_product_type_code` should FK to controlled vocabularies (`enum`, `product_type`) when semantics are locked.

---

## PRODUCT_TYPE ŌĆö What physical housing product is this?

**Asset classification** (SFR, BTR, MF garden, etc.); carries single-/multi-family flags and `vertical_code`.

**Enforcement:** Product-specific metrics and datasets should appear in **`bridge_product_type_metric`** or **`dataset_product_type`**. **`bridge_product_type_bedroom_type`** is the authoring bridge for bedroom applicability; **`product_type.bedroom_type_codes`** is machine-synced via `scripts/reference/catalog/sync_product_type_bedroom_codes_from_bridge.py` until the comma column is dropped. **`bedroom_type`** is the small vocabulary seed for FK tests.

---

## GEO_LEVEL ŌĆö At what geographic grain is this measured?

**Spatial resolution vocabulary** (property, ZIP, county, CBSA, state, national). Governs valid joins and rollups via **`grain_rank`** and capability flags (`fips_available`, `h3_available`, `code_system`).

**Enforcement:** Every `metric.geo_level_code` (and dataset where applicable) should exist in `geo_level`. Periodically verify all `geo_level` rows have `supported_in_reference` and `shape_expected` populated.

---

## VERTICAL ŌĆö Which business line?

Top-level classifier (**RESI**, **REQ**, **RED** in current seeds). `opco`, `offering`, `product_type`, and `concept_vertical` should all respect `vertical_code` FKs.

---

## DOMAIN ŌĆö What economic domain does this concept belong to?

**Semantic grouping** for routing and causal chains. In **this repo**, `domain.csv` defines **five** stable codes (not ŌĆ£demand / supply / macroŌĆØ labels): **capital**, **housing**, **household**, **place**, **portfolio** ŌĆö each with human and AI-oriented descriptions.

**Enforcement:** `concept.domain` ŌåÆ `domain.domain_code` is already enforced in dbt (`relationships` on `concept`). Optional future simplification: fold tiny stable domains into a unified `enum` pattern if maintenance cost warrants it.

---

## Summary: enforcement gaps to fix

| Layer | Gap | Intended fix |
|--------|-----|----------------|
| Warehouse | All columns `IS_NULLABLE = YES` on catalog tables | Human-applied DDL spec: **`ddl/catalog_constraints.sql`** (not dbt post-hooks) |
| DATASET | `last_refresh_date` typed as text in warehouse | `DATE`; stamp from export task; `NOT NULL` when active + MotherDuck |
| DATASET | `product_type_codes` comma string | **Bridge-authored**; comma column **derived** until follow-up PR drops it |
| VENDOR | `contract_expiry_date` stored as number | `DATE` |
| VENDOR | `is_motherduck_served` redundant with dataset | **Removed from `vendor` seed** ŌĆö use **dataset** only |
| METRIC | Raw vs built drift for active rows | **Singular test** on active + `data_status_code = active` |
| OPCO | `geo_status_code` unclear | Enum or drop |
| PRODUCT_TYPE | `bedroom_type_codes` comma string | **`bridge_product_type_bedroom_type`** + **`bedroom_type`** seed; comma column **derived** until drop |
| CONCEPT | Domain as free text in warehouse | Align column to `domain.domain_code` (dbt already tests FK on seed) |
| METRIC `unit` | Duplicated YAML allowlists | **`unit` seed** + **`dbt_utils.relationships_where`** on **metric** / **metric_raw** |
